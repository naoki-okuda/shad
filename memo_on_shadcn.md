# overrides機能

`custom_dir` は

> テーマのルートを上書きする仕組みです。
つまり：
* `templates/*.html` → 置き換え可能
* `partials/*.html` → 置き換え可能
* `assets/*.css` → 置き換え可能
* `assets/*.js` → 置き換え可能



# 具体的に何ができるか

## ✅ 1. レイアウト変更

例：

```jinja
main.html
base.html
```

を override すれば、

* ヘッダー構造変更
* サイドバー構造変更
* フッター変更
* メタ構造変更

すべて可能。

---

## ✅ 2. SEO完全制御

* JSON-LD自由設計
* OGP自由設計
* canonical制御
* robots制御

今回やったのがまさにそれです。

---

## ✅ 3. デザイン改変

* Tailwindクラス書き換え
* DOM構造変更
* ボタンやカードの追加

shadcnはReact由来のデザイン思想ですが、
MkDocs版は静的テンプレートなので普通にHTML改造可能です。

---

## ✅ 4. CSS完全上書き

```yaml
extra_css:
  - css/custom.css
```

よりも強く、

```text
overrides/assets/styles.css
```

でテーマのCSSそのものを置き換えられます。

---

# できないこと

## ❌ Pythonロジックは触れない

例えば：

* ナビ生成ロジック
* ページビルド処理
* Markdown処理

これはテーマではなく MkDocs 本体側。

---

## ❌ テーマの内部プラグイン処理

もしテーマが Python 側で何か処理していたら、
それは override できません。

（shadcnはほぼテンプレート主体なので問題なし）

---

# 実質どういう状態か

今あなたは：

> テーマをforkせずに
> テーマを改造できる状態

にいます。

これはかなり強いです。

---

# ただし注意

テーマをアップデートした場合：

* 元テーマのHTML構造が変わる
* override が古くなる

可能性があります。

つまり：

> override は「フォークに近い」

という認識が正確です。

---

# 技術的に言うと

MkDocs のテーマ解決順序は：

1. custom_dir
2. テーマパッケージ

なので、

override に同名ファイルがあれば完全に上書き。
